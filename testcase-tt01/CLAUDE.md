# 项目指引

本仓库为测试用例流水线项目，维护多端 AI 测试知识库。

## AI 测试知识库维护规范

- 知识库存放于 `ai-context/`，结构遵循 `ai-context/README.md` 的描述。
- 任何功能变更必须同步更新对应端模块的需求和接口文件。
- 文件格式必须符合 `.claude/skills/ai-context-maintainer/templates/` 中定义的模板。
- 在完成更新后，应执行完整性检查。

## 防幻觉铁律

维护知识库时为避免编造信息，执行以下两条铁律（详见 `.claude/skills/ai-context-maintainer/SKILL.md` §5.1）：

### 铁律 A · 高危内容逐项溯源

写入下列"高危内容"时必须**逐项标注来源**，不得编造、不得笼统标注、不得缺省：

- API 错误码表（每个错误码）
- 字段类型与约束（每个字段）
- 状态枚举值（每个枚举项）
- 具体数值（百分比、超时、额度上限、手续费、阈值等）
- 测试数据与测试账号

有效来源仅三类，标注格式：
- 文档：`（源：{文档名} §{章节}）`
- 代码：`（源：代码 {文件:行}）`
- 用户：`（源：用户提供 {YYYY-MM-DD}）`

高危项缺来源时：写 `待补充` 并注明"需提供什么"，**绝不以推断填充**。

### 铁律 B · 写前闸门

任何知识库写入前逐条自检，全部通过方可写入；任一不通过则该部分不写、转为待办输出：

1. 本批涉及的高危内容是否每一项都挂了有效来源？
2. 有无高危项因缺来源而被编造填充？（应为 `待补充`）
3. 推测性内容是否已标注 `（未证实）` 或置于"待确认"区，未混入事实正文？

非高危内容不强制逐项溯源，但仍遵守"不编造"底线。

## 知识库 skill 协作

知识库维护由两个 skill 协作完成：

- **`ai-context-ingest`**：读取/解析用户提供的文档，结构化提取并逐项溯源、标记高危内容、检测冲突，产出提取报告（仅内存呈现，不落盘、不写 `ai-context/`）。
- **`ai-context-maintainer`**：`ai-context/` 的**唯一写入方**。接收提取报告后按模板写入对应端/模块文件，执行写前闸门与完整性检查，更新 changelog。

协作链路：用户提供文档 → `/ai-context-ingest` 解析产报告 → `/ai-context-maintainer` 按报告写入。

分工原则：ingest 只提取不写入，maintainer 只写入不解析；写入/模板/防幻觉逻辑集中在 maintainer，不在两处重复。
