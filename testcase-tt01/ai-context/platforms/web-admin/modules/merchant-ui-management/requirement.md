# 商户端 UI 管理 需求说明

> 端：web-admin ｜ 模块：merchant-ui-management ｜ 对应表：`zs_scrm_banner`、`zs_scrm_contact_us`、`zs_scrm_feedback`、`sys_notice`

## 用户故事

- 作为平台运营，我希望管理 banner 轮播图（商家后台首页/登录页/客户端），以便发布运营推广内容。
- 作为平台运营，我希望维护"联系我们"配置（客服/电话/邮箱/Telegram/WhatsApp/二维码），以便商家与客户端用户联系平台。
- 作为平台运营，我希望接收并处理商家提交的意见反馈，以便改进产品。
- 作为平台运营，我希望发布系统公告/通知，以便向商家广播信息。

## 验收标准

- [ ] 场景：新建 banner，预期：填写标题、图片 URL、跳转链接、展示位置（1商家后台首页 2登录页 3客户端）、排序、起止时间、状态，落库 `zs_scrm_banner`。
- [ ] 场景：banner 定时上下线，预期：按 start_time/end_time 与 status 控制展示，过期自动下线。
- [ ] 场景：维护联系我们，预期：按渠道类型（1客服 2电话 3邮箱 4Telegram 5WhatsApp 6二维码）配置内容/二维码，`status` 控制显隐。
- [ ] 场景：处理意见反馈，预期：查看反馈（问题类型/描述/支持平台/联系方式/图片），维护 `handle_state`（0待处理 1处理中 2已处理 3已关闭）并填写平台回复 `reply`、处理人、处理时间。
- [ ] 场景：反馈图片上传校验，预期：最多 4 张，单张 ≤2M，jpg/png/jpeg，存 `images`(JSON)。
- [ ] 场景：发布系统公告，预期：落 `sys_notice`，商家后台/客户端可见。

## 业务规则

1. banner 按展示位置区分投放端（商家后台/登录页/客户端），受起止时间与状态控制。
2. 联系我们按渠道类型多行配置，`status` 控制显隐。
3. 意见反馈由商家（user_id）/员工（staff_id）提交，平台运营处理；问题类型分文字翻译/图片翻译/其他，问题描述限 200 字。
4. 反馈图片上限 4 张、单张 ≤2M、格式 jpg/png/jpeg。
5. 公告/通知为广播性质，无复杂双向交互。

## 约束与依赖

- 展示侧：web-merchant（商家后台首页/登录页）、pc-client（客户端）。
- 反馈数据来源：web-merchant / pc-client 用户提交。
- 接口/数据模型/UI/测试笔记：暂未提供，待补充（数据模型见 `shared/common-data-model.md` A5）。
