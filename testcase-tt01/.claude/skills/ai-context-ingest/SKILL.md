# ai-context-ingest (文档解析入库)

## 概述

专门负责"读文件 → 解析 → 结构化提取 → 溯源标注 → 高危标记 → 冲突检测 → 产出提取报告 → 移交 `ai-context-maintainer` 写入"。

**核心边界：本 skill 只提取、不写入。** `ai-context/` 的唯一写入方是 `ai-context-maintainer`；本 skill 产出提取报告（仅内存/直接呈现，不落盘），交由 maintainer 按模板写入。

**与 `ai-context-maintainer` 的分工：**

| 职责 | ai-context-ingest（本 skill） | ai-context-maintainer |
|---|---|---|
| 读文件/解析 | ✅ | ❌ |
| 文档类型/端/模块识别 | ✅ | ❌ |
| 结构化提取 + 溯源标注 | ✅ | ❌ |
| 高危内容标记（铁律 A） | ✅ 标记 | ✅ 写入时执行 |
| 冲突检测 | ✅ | ✅ 处理 |
| 按模板写入 5 文件 | ❌ | ✅ |
| 完整性/规范性检查 | ❌ | ✅ |
| changelog | ❌ | ✅ |

## Instructions

### 1. 输入与触发

- **入参**：文件或文件夹路径（必填）+ 可选意图提示（端/模块名）。
- 触发场景：用户提供文档路径并要求"更新知识库 / 解析入库 / 读进去"等。
- 若未提供路径，先向用户索要；不接受无源文件的纯口述作为提取来源（口述可作为"用户提供"来源，但须标注 `（源：用户提供 {YYYY-MM-DD}）`）。

### 2. 处理流水线

1. **读取**：读取目标文件全文；超大文件按章节分段读取，避免截断丢内容。
2. **文档分类**：识别类型（见 §3，详见 `references/doc-type-rules.md`）——数据库设计文档 / 模块需求说明 / 项目描述 / 接口文档 / 其他。
3. **端/模块定位**：按文档类型路由目标端与模块（见 §4）。**自动识别为主，置信度低或歧义时列候选问用户**，不擅自假定。
4. **结构化提取**：按目标端的 5 文件维度（requirement/api/data-model/ui-spec/test-notes）提取事实；每条事实挂来源 `（源：{文件名} §{章节}）`。
5. **高危标记**：按铁律 A 标记字段类型/枚举/数值/错误码/测试数据，逐项溯源；缺来源标 `待补充` + 注明需提供什么（见 §5）。
6. **冲突检测**：与现有 `ai-context/` 内容比对，发现冲突列入冲突清单，**不裁决**（见 §6）。
7. **产出提取报告**：按 `references/report-schema.md` 模板组织，直接在对话中呈现（不落盘）。
8. **写前闸门自检**：对提取报告执行铁律 B 三项自检（高危是否逐项溯源 / 有无编造填充 / 推测是否隔离），全过才移交。
9. **移交**：使用 Skill 工具调用 `ai-context-maintainer`，将提取报告作为参数传入，由其按模板写入；或提示用户调用 `/ai-context-maintainer` 并附报告。

### 3. 文档分类

按内容特征判定（详细策略见 `references/doc-type-rules.md`）：

| 类型 | 识别特征 | 提取重点 |
|---|---|---|
| 数据库设计文档 | 含 CREATE TABLE / 表清单 / ER 图 / 字段定义 | 实体字段(类型/约束/说明)、状态流转、ER 关系、设计约定、关键设计说明 |
| 模块需求说明 | 标题含"模块说明"，描述页面/字段/校验/操作 | 5 文件全维度：用户故事/验收/规则、接口、实体字段、UI 布局/校验/异常文案、缺陷/回归/测试数据 |
| 项目描述 | 标题含"项目描述/背景"，描述端定位与流程 | 端简介/技术栈/交互范式/模块职责/端到端流程/与其他端关系 → platform-overview |
| 接口文档 | 含接口路径/请求响应/错误码 | 路径/方法/参数表/响应示例/错误码表 → 模块 api.md |
| 其他 | 不属于上述 | 列出可识别信息，问用户归属 |

### 4. 端/模块识别规则

- **数据库设计文档**：按域拆分——A 域 → web-admin / shared，B 域 → web-merchant，C 域 → pc-client；公共字典与设计约定 → shared。一张表可能被多端引用，以"维护方"端为主写入、其余端引用。
- **模块需求说明**：从标题/正文识别归属端关键词——"商户端/商家后台" → web-merchant，"管理端/管理后台/平台运营" → web-admin，"客户端/桌面/PC 端" → pc-client；模块名取标题中的模块名并转 kebab-case（如"激活码管理" → activation-code-management）。端或模块任一不明确 → 列候选问用户。
- **项目描述**：按标题/内容识别端（桌面应用 → pc-client，web 商家后台 → web-merchant，管理后台 → web-admin），写入对应 `platform-overview.md`。
- **接口文档**：需用户指明归属端/模块；无提示则按接口名匹配现有模块，匹配不到则问用户。
- **跨端文档**：一份文档覆盖多端/多模块时，按端/模块拆成多份提取报告分别移交。

### 5. 提取规则与溯源标注

- **每条事实必挂来源**：`（源：{文件名} §{章节}）` / `（源：代码 {文件:行}）` / `（源：用户提供 {YYYY-MM-DD}）`。
- **高危内容逐项溯源**（铁律 A，逐项不可笼统）：
  - API 错误码表 —— 每个错误码
  - 字段类型与约束 —— 每个字段
  - 状态枚举值 —— 每个枚举项
  - 具体数值 —— 百分比/超时/额度上限/手续费/阈值/字符长度等
  - 测试数据与测试账号
- **缺来源处理**：写 `待补充` 并注明"需提供什么"，**绝不以推断填充**。
- **推测隔离**：凡属推断/融合/猜测，标 `（未证实）` 并置于报告"冲突/待确认"区，不混入事实正文。
- **不编造**：文档未提及的内容（如模块说明里没有接口参数）一律 `待补充`，不得自行补全。

### 6. 冲突检测

- 比对提取内容与现有 `ai-context/`（含 shared、各端模块、glossary、system-overview 等）。
- 命名不一致（如 SCEM vs NexSCRM）、规则冲突（如 web商家后台 vs 迭代001）、字段语义出入等 → 列入报告"冲突清单"，编号 `C1/C2…`，标注双方来源。
- **不裁决、不自动选择一方**；移交时提示 maintainer 按 `（未证实）` 处理并向用户确认。

### 7. 移交

- 提取报告在对话中完整呈现后，调用 Skill 工具 `ai-context-maintainer`，传入报告与"按提取报告写入"任务意图。
- maintainer 负责：按报告内容写入对应端/模块文件（模板填充）、执行铁律 B 写前闸门、完整性检查、更新 changelog。
- 若报告含冲突/待确认项，移交前先就关键冲突向用户确认；确认结果回填报告再移交。

### 8. 行为约束

- **不写入 `ai-context/`**：本 skill 仅产提取报告，任何文件写入由 maintainer 执行。
- **不落盘中间产物**：提取报告仅内存/对话呈现，不写临时文件。
- **只基于源文件提取**，绝不编造业务规则、字段、接口、数值。
- **不裁决冲突**：只列出，等用户。
- **保持格式一致**：提取报告遵循 `references/report-schema.md`；事实陈述与现有知识库 Markdown 风格对齐。
- **每次产出后输出摘要**：源文件、类型、目标端/模块、高危项数、冲突项数、移交状态。

### 9. 依赖

- 依赖 `ai-context-maintainer` 已初始化的 `ai-context/` 结构与模板（`.claude/skills/ai-context-maintainer/templates/`）。
- 继承项目根 `CLAUDE.md` 的防幻觉铁律（铁律 A/B）。

## 注意事项

- 所有路径相对于项目根目录。
- 本 skill 不修改业务代码、不写 `ai-context/`。
- 大文档（如数据库设计文档）可按 A/B/C 域分段提取，必要时分多次移交。
- 提取报告是 ingest→maintainer 的契约，格式须严格，便于 maintainer 直接据此写入。
