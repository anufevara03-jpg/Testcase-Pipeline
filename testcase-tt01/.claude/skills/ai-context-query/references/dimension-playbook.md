# 可测维度组织法（场景 B 核心）

> 场景 B 把加载的模块知识按 **6 个可测维度**重组，让下游 `biz-test-analysis` / `testcase-create` 直接拿到"按用例类型分好类、带溯源"的上下文。每个维度注明 KB 来源与对应的用例类型。

## 6 维度 ↔ KB 来源 ↔ 用例类型

| 维度 | KB 来源 | 驱动的用例类型 |
|---|---|---|
| **主流程** | requirement 用户故事/验收标准 + ui-spec 页面跳转/联动规则 + common-business-rules 适用规则 | 冒烟用例（P0）、主流程正向用例 |
| **边界** | data-model 字段约束数值（长度/上限）+ ui-spec 校验规则的数值边界 | 边界值用例（刚好/越界） |
| **枚举** | data-model 状态枚举 + api 参数枚举值 + common-data-model §一.6 枚举约定 | 枚举遍历用例（每个枚举项） |
| **校验** | ui-spec 前端校验规则 + api 错误码 + 业务规则（重复/唯一/必填/格式） | 输入校验用例、拦截用例 |
| **异常** | test-notes 已知缺陷 + api 错误码 + 状态流转非法跳转 + 业务规则失效模式 | 异常场景用例、负面用例 |
| **回归** | test-notes 回归测试重点 + common-business-rules 跨端规则 | 回归用例、易错点用例 |

## 逐维度组织法（含 customer-profile-management 实例）

> 以下实例全部取自 `web-merchant/customer-profile-management` 真实内容，展示如何把 KB 事实归入维度。

### 主流程
- 来源：`requirement.md` 验收标准（`- [ ]` 列表）、`ui-spec.md` 页面联动规则。
- 组织：每条验收标准 → 一条主流程用例；联动规则补跨端预期。
- 实例：
  - 新增字段后，后台客户详情和 PC 客户画像面板立即展示，历史客户显示空值（源：requirement.md 验收标准 / `req-AC-1`）
  - 编辑字段（名称/选项/分组）后两端表单同步更新，已保存值保留（`req-AC-2`）
  - 删除/隐藏字段后两端不展示但历史值保留（`req-AC-3`）
  - 分组排序变更后两端展示顺序同步（`req-AC-4`）

### 边界
- 来源：`data-model.md` 数值约束、`ui-spec.md` 校验规则的数值项。
- 组织：每个数值约束 → 刚好/越界两类用例。
- 实例：
  - 自定义字段名最长 **10** 字 → 10 字通过 / 11 字拦截（源：data-model.md §四§1 / `dim:boundary:fieldNameLength`）
  - 自定义分组名最长 **30** 字 → 30 字通过 / 31 字拦截（`dim:boundary:groupNameLength`）

### 枚举
- 来源：`data-model.md` 状态枚举、`api.md` 参数枚举、`common-data-model.md §一.6`（枚举用 TINYINT+注释）。
- 组织：每个枚举项 → 一条遍历用例；状态流转 → 合法跳转用例。
- 实例：
  - 字段类型 6 种（文本/多行文本/日期/单选/下拉单选/下拉多选）→ 各创一条字段（源：data-model.md §四§2）
  - 客户价值枚举（一般/重要/核心/其他）、性别（男/女/未知）、活跃度（已流失/低/中/高活跃）→ 下拉单选遍历
  - 字段状态流转：显示 ↔ 隐藏（源：data-model.md 状态流转）
  - > 对比模块 `activation-code-management` 的 `range_type`（去重范围 1-6，6 个枚举项）、`port_alloc_type`（1动态/2固定）—— 列类型完整的模块，枚举用例可完整生成。

### 校验
- 来源：`ui-spec.md` 前端校验规则表、`api.md` 错误码、业务规则（重复/唯一/必填）。
- 组织：每条校验规则 → 触发拦截用例；预期值用精确文案。
- 实例：
  - 分组名称重复 → 提示"分组名称已存在"（源：ui-spec.md 校验规则表）
  - 字段名称重复（跨分组唯一）→ 提示"字段名称已存在"
  - 删除含字段的分组 → 拦截"分组内存在自定义字段，请先删除字段后再删除分组"
  - > api 错误码本模块全 `待补充` → 校验类用例的 API 层预期值占位"待产品确认"，**不编造错误码**。

### 异常
- 来源：`test-notes.md` 已知缺陷、`api.md` 错误码、状态流转非法跳转、业务规则失效模式。
- 组织：缺陷 → 复现用例；非法状态跳转 → 负面用例；二次确认 → 确认/取消两条。
- 实例：
  - 删除分组二次确认"确认删除该分组吗？"（源：ui-spec.md 关键交互文案）
  - 删除字段二次确认"确认删除吗？删除后，所有包含此选项的内容将全部删除。"
  - 删除字段后历史值保留；再次添加该字段时历史值恢复显示（异常态恢复）
  - > 已知缺陷：本模块 `test-notes.md` 记录"暂无记录" → 异常用例主要来自状态流转与校验失效，而非历史缺陷。

### 回归
- 来源：`test-notes.md` 回归测试重点（已按场景分组）、`common-business-rules.md` 跨端规则。
- 组织：每组回归重点 → 至少 1 条回归用例；跨端规则 → 跨端联动用例。
- 实例（本模块 test-notes 已分 6 组）：
  - 字段管理回归：新增/编辑/删除/隐藏/拖拽排序
  - 分组管理回归：新增/编辑/删除/拖拽排序
  - 系统默认保护回归：默认分组及字段不可删/改/藏，仅可排序
  - 预览功能回归：右侧预览实时模拟 + 修改后即时刷新
  - 跨端联动：所有字段/分组变更须后台客户详情 ↔ PC 客户画像面板**两端同步**

## 使用要点

1. **先齐后分**：先按 §1 完整度检查确认 5 文件齐不齐，缺失的进"缺口"区，不要假装有。
2. **维度非互斥**：一条事实可进多个维度（如"字段名 10 字限制"既进边界也进校验）。按"主要驱动用例类型"归主维度，其余维度交叉引用。
3. **缺口优先暴露**：`待补充` 项（尤其错误码/列类型/测试数据）无论落在哪个维度，都要**额外汇总到独立的"缺口清单"**，这是下游用例"待产品确认"的来源。
4. **保留溯源与锚点**：每条事实挂 `（源：…）` 与 `kb_anchor`（见 `output-schema.md`），供下游 coverage 回溯。
