# KB 上下文包模板（query → biz-test-analysis 契约）

> `ai-context-query` 产出的 KB 上下文包按本模板组织，在对话中直接呈现（**不落盘**），交由 `biz-test-analysis` / `testcase-create` 消费。
> 下游读取本包后：按可测维度直接设计用例（无需再从原文档提取）、每条用例结论挂 KB 来源、`待补充` 项转为"待产品确认"占位、`coverage.json` 用 `kb_anchor` 回溯。

## 通用规则

- 所有 `{...}` 占位在最终输出替换为实际内容或 `待补充`，不得保留裸占位、不得编造。
- **溯源标注原样保留 KB 内层来源**：KB 中的事实本就带 `（源：{原文档名} §{章节}）`（如 `（源：客户画像管理模块.md §一）`），**原样保留原文档名，不改写为 KB 文件名**；需标注读取自哪个 KB 文件时，用"经 {kb-file}"后缀，如 `（源：客户画像管理模块.md §一，经 requirement.md）`。同时挂 `kb_anchor`（锚点形态见 SKILL.md §6）。
- `待补充` 项一律进"四、缺口清单"，**不计入下游覆盖率分母**；下游对应预期值占位"待产品确认基准值"。
- `（未证实）`/`待确认` 原样透传，不裁决。

## 使用说明

- 上下文包在对话中完整呈现，不写入磁盘。
- 场景 B 是 `biz-test-analysis` 的**首选输入**：下游见本包结构（6 维度 + 溯源 + 锚点）即走 KB 链路；否则降级读原始文档并提示"建议先入库"。
- `biz-test-analysis` 据本包设计用例时：覆盖清单（FP-xxx）从本包维度派生，每个 FP 挂对应 `kb_anchor`；`testcase-create` 据此生成 `coverage.json`（每项含 `kb_anchor` 字段）。
- 缺口清单的项在 `testcase-create` 第 2 步作为"待澄清问题"逐条问用户；用户给不出 → 预期值占位"待产品确认基准值"。
- 一端一模块对应一个上下文包；不混装多模块。
