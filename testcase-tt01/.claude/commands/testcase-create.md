# 创建测试用例（业务分析 → Excel 用例）

为指定业务模块生成一份可在集成测试环境直接执行的测试用例集（Excel），遵循「**先确认缺口 → 再冒烟 → 后完整**」原则，且需求点/功能点覆盖率 ≥90%。

## 需要的三项输入

| 输入     | 必需 | 说明                                                                   |
| -------- | ---- | ---------------------------------------------------------------------- |
| 项目描述 | ✅   | 系统定位、角色、核心业务流程、架构。可内联文字，或给文件路径(.txt/.md) |
| 模块描述 | ✅   | 本次覆盖的具体模块/功能点需求。可内联文字，或给文件路径                |
| 输出目录 | ✅   | Excel 用例文件保存目录（绝对或相对路径）                               |

> 用户给的是文件路径时，先用 Read 读取内容再分析，不要让用户重复粘贴文本。

## 沟通风格

- 输出结构清晰、专业，解释详尽，像在给高级测试工程师做培训。
- 信息不足以支撑全局分析时，主动反问索要必要上下文（架构图/接口文档/业务规则/计费规则），而不是勉强给出模棱两可的用例。
- 始终以「交付一个能在集成测试环境直接执行的测试集」为目标，而不只是灵感建议。
- 期望值若依赖未明确的业务规则（如计费公式、阈值、上限），**不要编造数字**，标注「待产品确认基准值」。

## 工作流

### 第 0 步：一次性收齐输入

检查三项输入是否齐备，**把所有缺失项一次性问清**（结构化提问或 AskUserQuestion），避免逐条追问。

- 项目描述缺失 → 索要（或提示可用 `biz-test-analysis` 先分析）
- 模块描述缺失 → 索要
- 输出目录缺失 → 索要；若用户给了模块描述所在目录，可建议其作为输出目录并请用户确认

### 第 1 步：业务测试分析

调用 skill `biz-test-analysis`，基于项目描述 + 模块描述输出：系统分析、测试关注点、核心场景（主流程）、异常场景、高频风险、风险分级、待澄清问题。本阶段只分析，不写用例。

> 产出里务必带出**覆盖清单草案**：把模块拆成可测的需求点/功能点（每点赋 ID `FP-001`…），作为后续覆盖率校验的分母；同时整理**待澄清问题清单**供第 2 步确认。

### 第 2 步：需求缺口逐条确认（在生成用例之前）

**先确认、再写用例**——避免写出后又因规则不明返工。把第 1 步的「待澄清问题清单」逐条与用户确认，拿到确定规则后再进入设计；无法确认的保持「待产品确认」，设计时对应预期值占位、**不编造数字**。

1. **一次性呈现清单**：每条写清「问题 + 涉及哪个需求点 + 不确认会阻塞哪类预期值」。
2. **逐条确认**：请用户就该条回复——
   - **提供**：给出明确的业务规则/数值/阈值（自由文本）。
   - **略过**：该条保持「待产品确认」标记，不强行编造。
   - 允许一次性批量回复（如「Q1:…，Q2 略过，Q3:…」）以减少往返；逐条 `AskUserQuestion` 仅在某条本身是离散选项（如「阈值 `<10000` 还是 `≤10000`」）时使用，纯数值/规则类问题用自然语言更直接。
3. **记录确认结果**：确认到的规则用于第 3 步设计；略过的标「待产品确认」。

> 设计用例过程中若浮现新的歧义/缺口，回到本步补充确认，再继续设计。清单为空则直接跳过本步。

### 第 3 步：设计并生成 Excel 用例

调用 skill `excel-test-case-generator`，按以下约定生成：

- **建覆盖清单**：把第 1 步的覆盖清单草案落成 `coverage.json`（`[{"id","item"}]`），设计用例时每条用 `covers` 标注覆盖的清单项 ID，确保每项至少 1 条用例覆盖（目标 ≥90%）。
- **冒烟优先**：先 5-15 条核心主流程（全 P0，标记 `smoke: true`），再产出完整用例。
- **单文件双 Sheet**：冒烟用例 Sheet + 完整用例 Sheet，放进同一个 Excel。
- **列**：用例编号、用例名称、所属模块、优先级、前置条件、测试步骤、预期结果、备注。
- **编号**：冒烟 `SMK-<模块>-NNN`，完整 `TC-<模块>-<子类>-NNN`（自定义有意义编号）。
- 技能自带 CLI 默认输出两个文件、自动 TC-001 编号、无「所属模块/备注」列；要达成上述约定，需写一个复用技能 `TestCaseGenerator` 样式的构建脚本（单文件双 Sheet + 自定义列与编号 + 优先级颜色标记），生成后删除临时脚本与中间 JSON。
- 先确认 `python` 与 `openpyxl` 可用再生成。

### 第 4 步：质量自检

用技能 CLI 跑自检，**全部达标再交付**：

- `--check-vague`（模糊表述）、`--check-duplicates`（重复用例）、`--check-coverage --coverage-list coverage.json`（**覆盖率 ≥90%**），并核对 P0 占比（建议 25%-30%）。
- 预期结果中的模糊词（一致/正确/成功/正常/相关/相应/完整/适当等）改为可验证描述（用「= 数值相等」「文本为X」「跳转到 /xxx」）。
- 甄别并容忍误报（如产品名 Grok 触发 ok 子串匹配）。
- 覆盖率 <90% → 按未覆盖项补用例；P0 超出区间则调整（纯 UI 展示不应设 P0）。

### 第 5 步：校验与交付

- 校验 Excel：双 Sheet、行数、表头、文件可正常打开。
- 生成质量自检报告（.txt，可用 `--quality-report --coverage-list` 带出覆盖率段落）作为交付文档。
- 清理临时文件（中间 JSON、构建脚本），仅保留 .xlsx 与报告。
- 向用户报告：成功/失败、路径、用例构成（冒烟/完整、优先级分布、覆盖的功能域）、**覆盖率与未覆盖项**、**需求缺口确认状态**（已确认/仍待确认逐条标注）。

> 第 2 步被用户略过的条目一律保持「待产品确认」原样，绝不臆测数值。若交付后用户补充了新规则，回到第 3 步改进受波及用例并重跑第 4 步自检。

## 失败处理

任一步失败：明确报错原因、已完成步骤、建议修复动作，不要静默继续。
